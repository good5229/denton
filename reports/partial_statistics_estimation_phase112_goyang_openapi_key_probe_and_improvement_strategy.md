# Phase112 고양시 OpenAPI key 확인 및 성능개선 전략

## key 확인

`.env`의 `DATA_GOYANG_KEY` 존재를 확인했다. 값은 출력하지 않았다.

Python 기본 SSL 검증에서는 고양시 서버 인증서 체인 문제로 실패했지만, `curl -k` 호출에서는 고양시 OpenAPI가 정상 JSON을 반환했다. 따라서 key 자체는 유효하다.

## 왜 바로 성능 개선으로 반영하면 안 되는가

고양시 OpenAPI는 행정동·좌표·시설규모 자료로서 공간 해상도 개선에는 강하지만, 대부분 “현재 스냅샷”이다. 따라서 2023년 중분류 GVA를 예측했다는 성능 주장에 바로 넣으면 공표시점 문제가 생길 수 있다.

즉, 이 자료는 다음 둘 중 하나로 써야 한다.

1. **정밀화 지표**: 사후 이용 가능한 공간·시설 구조로 2023년 또는 2024년 GVA를 더 정밀 배분.
2. **속보 후보**: 동일 API의 과거 기준월·갱신일·원자료 빈티지가 확인될 때만 속보 성능값에 반영.

## 성능 개선에 적합한 새 방법

기존의 단일 일반지표 방식은 남은 취약 업종에 맞지 않는다. 앞으로는 `업종군별 지표 묶음 + 제약형 가중치 선택`이 필요하다.

### 1. 업종군별 지표 묶음

| 취약 업종군 | 후보 지표 |
|---|---|
| 스포츠·오락 서비스업 | 체육시설 수, 시설면적/관람석/운영주체, 공원면적, 관광·문화시설 |
| 수도·하수·폐기물 | 하수처리시설 처리량·용량, 쓰레기 배출/처리량, 환경오염 배출사업장 |
| 부동산·사업시설 | 건축물 용도별 연면적, 아파트 실거래금액, 주차장/상업시설 면적 |
| 보건·사회복지 | 병의원 수, 병상 수, 의료기관 면적, 복지시설 |
| 제조업 | 제조업 중분류 종사자/사업체, 출하액·부가가치 사후통계, 공장면적·전력 |
| 정보통신·방송·콘텐츠 | 방송/언론매체, 콘텐츠 기업·사업장, 서버/플랫폼 활동지표 |
| 협회·단체 | 비영리단체 등록, 지방보조금, 단체 예산·회원·사업규모 |

### 2. 가중치 선택 방식

단일 후보를 채택/기각하는 방식보다 다음 방식이 낫다.

- 상위산업별로 후보지표를 여러 개 만든다.
- 2021~2024처럼 실제 중분류 검증이 가능한 기간에서 **연도 차단 교차검증**을 한다.
- 가중치는 0 이상, 합계 1 이하로 제한한다.
- 기존 추정값을 항상 포함시켜 새 지표가 약하면 자동으로 기존값에 수렴하게 한다.
- 평가 기준은 하나가 아니라 `총오차`, `10% 초과 셀`, `고액 업종 오차`, `최대 악화폭`을 동시에 본다.

### 3. 엄격 기준 완화 방향

이전의 “개별 중분류 하나라도 악화되면 미채택” 기준은 너무 보수적이라 실제 개선을 막는다. 대신 다음 기준이 현실적이다.

- 상위산업 WAPE가 감소해야 한다.
- 10% 초과 중분류 수가 증가하면 안 된다.
- 실제 GVA 비중이 큰 중분류의 오차가 악화되면 안 된다.
- 소액 중분류 악화는 허용하되, 최대 악화액과 최대 악화율 상한을 둔다.

이렇게 해야 포항 제조업처럼 총오차가 크게 줄어드는 후보를 무조건 버리지 않고, 실질 개선 후보로 다룰 수 있다.

## 다음 실행 제안

다음 단계는 고양시 OpenAPI 자료를 이용해 `정밀화 전용` 실험을 먼저 하는 것이다.

1. 고양시 OpenAPI의 시설/건축/실거래 자료를 행정동에 붙인다.
2. 취약 업종별 후보지표 테이블을 만든다.
3. 상위산업별 제약형 가중치 탐색을 실행한다.
4. 기존 엄격기준과 완화기준을 나란히 보고, 어느 기준에서 실제 성능 개선이 발생하는지 비교한다.
5. 속보오차에는 아직 반영하지 않고, 정밀오차 개선 실험으로만 보고한다.
